This document provides a complete description of the workflows and workflow nodes created for this publication.

Workflows

Three different workflows are published called: RetroPath2.0, RP Analysis and rpRanker. We separate the tools that perform the retrosynthesis and the ones that performs the analysis into two different workflows to open the possibility of the workflows to be used with different algorithms. For example, development of RetroPath3.0 could mean that one may use the new, or any other retrosynthesis algorithm in place of RetroPath2.0 included here, while still using the collection of analysis tools to evaluate the generated pathways. The rpRanker workflow is a combination of the two aforementioned workflows, with a few other tools that will be described below.

rpRanker

The workflow is a combination of the RetroPath2.0 and the RP Analysis workflows and can be found here: https://synbiocad.micalis.inra.fr/mssb/u/mel_admin/w/rpranker (shown in Figure 1). To run it, a user must simply provide the target molecule in the form of an InChI description of the structure, an GEM SBML that the target molecule will be expressed in and the maximal number of enzymatic steps the user would like to explore. Note that the user may provide his own list of reaction rules, but if left blank the tool work with a default list of reaction rules from https://retrorules.org/.

Figure 1: rpRanker workflow as seen in the Galaxy platform. 1) rpRanker Galaxy UI to run the workflow. 2) The RetroPath2.0 and RP Analysis workflows are combined to perform a full analysis and RP Report parses the output to a user-friendly CSV format.

RetroPath2.0

The workflow may be found at the following location: https://synbiocad.micalis.inra.fr/mssb/u/mel_admin/w/retropath2 and includes three main nodes: RetroPath2.0, RP2paths and RP2 Reader. The Retropath2.0 workflow is shown in Figure 2 and requires the same information as the rpRanker workflow needs to run it. Tools are executed in the following order: “Sink from SBML” node takes for input an SBML file with MetaNetX annotations where the results are parsed into a RetroPath2.0 friendly CSV format, while the “Make Source” node takes as input an InChI and outputs a CSV file. Both results are passed to the RetroPath2.0 node to generate a network of the potential metabolic routes that connects a target molecule to all the possible chemical species in the SBML model. That graph is then passed to RP2paths to enumerate the all the individual pathways. Lastly, the results are passed to the RP2 Reader that parses the distinct metabolic paths to individual SBML files. The output is a collection of enriched SBML files compressed within a TAR file that we denote rpSBML.

Figure 2: RetroPath2.0 workflow as seen in the Galaxy platform. 1) RetroPath2.0 Galaxy GUI to run the workflow. 2) Galaxy interface of the workflow description.

Figure 3: Example of a valid chemical description of an SBML. The workflow requires the MIRIAM annotations description of chemical species to contain MetaNetX ID’s.

RP Analysis

The workflow may be found at the following location: https://synbiocad.micalis.inra.fr/mssb/u/mel_admin/w/rp-analysis and includes the following nodes: rpCofactors, rpFBA, rpThermo and rpSelenzyme. A more complete description of the tools can be found in the next section. In this workflow, the user must provide a TAR file containing a collection of rpSBML files. The workflow performs its analysis and appends the results to the individual files within the TAR before outputting the results.

Figure 4: RP Analysis workflow as seen in the Galaxy platform. 1) RP Analysis Galaxy GUI to run the workflow. 2) The main nodes are the rpCofactors, rpFBA, rpThermo and rpSelenzyme that all perform a different type of analysis each (see descriptions of node) and write the results to the SBML files. The user must specify an SBML file for the rpFBA node that contains a full model, and will be the SBML file that will be merged to before performing that step. The user has the ability to retain the merged version or the single pathway version that was originally input.

Workflow Nodes

Sink from SBML

This node takes for input an SBML file, parses all the species within it and uses the MIRIAM annotations to identify the species to retrieve their InChI. The results are written to a RetroPath2.0 friendly CSV file format. In the advanced options one can specify the compartment from which the tool will extract the species from. The default is MNXC3, the MetaNetX code for the cytoplasm. If the user wishes to upload an SBML file from another source, then this value must be changed.

RetroPath2.0

The node takes for input a sink, a source file and a maximal pathway length. The advanced options include the optional parameter “rules file”, if not specified an internal list is used that is taken from RetroRules (https://retrorules.org/). The TopX, Minimum rule diameter, Maximal rule diameter, mwmax source and mwmax cof, parameters are RetroPath2.0 specific parameters whose more complete descriptions can be found here: https://www.myexperiment.org/workflows/4987.html. The Server URL is the IP address of the tools. In the above figure, a local address but external servers may be specified here. Lastly, the Timeout parameter can be modified to increase the execution time of the tool. A limit is set for the server version of the tool, that expects a large number of users. Another internal limit is set for the RAM usage upon deployment. Again, this is designed to limit the potential combinatorial explosion that may occur with RetroPath2.0 and enable the tool to be deployed on a server for numerous users to access it simultaneously. The output of the tool is a CSV file that describes the connection between the source and sink through their different metabolic routes. For a more complete description of the output please refer to the RetroPath2.0 documentation above.

Figure 10: RetroPath2.0 Galaxy node. The node takes for input a sink and a source file and the maximal pathway length. The advanced options (that are not required) include a series of RetroPath2.0 specific parameters. A rule files input that if not selected uses an internal list of reaction rules is used.

RP2paths

The node takes for input the output of RetroPath2.0 and enumerates the individual pathways. For a complete description of the tools please refer to the documentation from the original tools: https://github.com/brsynth/rp2paths. The advanced options include a “time out” parameter in seconds to limit the execution time of the tool. Similar to the RetroPath2.0 node, internal limit is set on RAM usage for this server version of the tool. The server URL is the IP address of the tool, here localhost on port number 5008.

Figure 11: RP2paths Galaxy node. The node takes for input the output from RetroPath2.0 and outputs the CSV file containing a list of compounds and a CSV file with the individual pathways.

RP2 Reader

The node takes for input the output of RetroPath2.0 and RP2paths. Because more than one reaction rule may be associated with a single step, the advanced parameters include the “out_paths max number of substeps per step”, that controls the Top X number of rules that will be used to calculate the number of sub-steps of a pathway. The SBML compartment ID determines the name of the compartment in which the pathway will be added to, with a default of “MNXC3” that corresponds to the cytoplasm for MetaNetX models. The “rpSBML path ID” parameter is the ID given to the heterologous pathway. Note that this ID will be used throughout the different tools to uniquely identify the heterologous species and reactions. The IP address is the address of the web service version of this tool.

Indeed, due to the nature of the calculation of chemical transformations by RetroPath2.0, more than one reaction rule may be associated with one reaction. When converting these to SBML, the best scoring reactions steps are considered before combining them to individual pathways (see additional information). The number of combinations is controlled by the “out_paths max number of substeps per step” parameter in the advanced options of RP2 Reader (see additional information). Careful consideration must be done when controlling that number since a combinatorial explosion may occur when choosing to high a number for a long pathway since:

The default value is 2 but may be restricted to a minimum of 1 with no maximum. Higher number may be included to grow the potential size of the different pathways. The option is left to the user as to what are the best parameterisation of this step, but combinatorial explosion may occur if many rules are associated with many steps in a pathway.

Figure 12: RP2 Reader Galaxy node. The node takes for input the output of RetroPath2.0 and RP2paths and parses the pathways into SBML files.

Figure 13: Illustration of RP2 Reader converting the output of RP2paths to a collection of SBML files with the heterologous pathways stored within. Highlighted in red is a case where multiple rules are associated with a single reaction. Under such a condition, the tool calculates all the possible sub-paths of the pathway.

RP Cofactors

The node takes for input a TAR collection of rpSBML files and adds the cofactors associated with each mono-component reactions generated by RetroPath2.0. The advanced parameters include the ID of the heterologous pathway defined by RP2 Reader, the compartment ID of the SBML file (default is MNXC3, the MetaNetX ID for the cytoplasm). Lastly the IP address of the tool is required for the web service version of that tool. The tool also updates the reaction rule to include the SMILES cofactors to the reaction rule.

Figure 14: RP Cofactors Galaxy node.

Figure 15: Example of cofactors (green) being added to a pathway calculated by RetroPath2.0. In the dashed box is an example of a reaction with its reaction rule with the added cofactors (green) added to the rule itself when these are added to the SBML file.

RP Thermodynamics

The node takes for input a collection of rpSBML files inside a TAR file. The tool calculates the thermodynamic feasibility using a modified version of the component analysis package (https://gitlab.com/elad.noor/component-contribution), that accepts InChI structures as well as MetaNetX and KEGG ID’s. The tool requires the ID of the heterologous pathway to be specified and the IP address of the web service version of the tool.

Figure 16: RP Thermodynamics Galaxy node. The user must provide a collection of rpSBML files within a TAR file.

Figure 17: Illustration of the Gibbs free energy calculation using component the contribution package. The arrows as colour coded from red (best) to blue (worst) to show the best and worst performing reactions.

Figure 17: rpSBML example of the annotations being added to a reaction under the BRSynth tag.

RP FBA

This node takes for input a TAR collection of rpSBML files and a SBML model file. Each heterologous pathway is individually added to the SBML model and FBA is performed using the FBC package. The results are a collection of rpSBML files, with the tags added as per the Figure 14 example. The user may return the merged version of the model, or the heterologous pathway only through the “Don’t output the merged model” boolean parameter. The tool also takes as advanced parameters the name of the heterologous pathway as created by rpReader and the compartment ID where the heterologous pathway exists in.

Lastly the tool contains the ability to create new reactions that creates a flux for an orphan species. That is, it is possible that some cofactors are not contained within full GEM model and thus, if required for a reaction, would lead to a flux of 0. To simulate for the condition where the cofactor is added to the growing environment of the cell we add a reaction, not unlike glucose, that creates the species. The default is set to “No” since it may be a filter to the users to select for a better pathway that do not need for such a consideration.

Figure 18: RP FBA Galaxy node. The tool takes as input a collection of rpSBML files and a full model SBML file.

Figure 19: Example of a reaction from a rpSBML file. The results of FBA are stored under the BRSynth tag for each reaction, and for every objective set by the FBC tag.

Figure 20: Illustration of the steps in the tool's calculation of FBA. The pathway is merged with a GEM SBML model and using the FBC package and CobraPy FBA is performed and the fluxes saved to the SBML file. The arrow size shows the individual step fluxes.

RP Selenzyme

This tool takes as input a collection of rpSBML files, that contains for each heterologous reaction a reaction rule SMILES. These reaction rules are updated with a ranked list of UniProt identifiers for associated enzyme sequences using Selenzyme. For more information please refer to the creator's website http://selenzyme.synbiochem.co.uk/.

Figure 20: RP Selenzyme Galaxy node.

Figure 18: Example of a reaction in a rpSBML file with Selenzyme information included. We add the UniProt ID to the MIRIAM annotations, and the Selenzyme score with the UniProt ID in the BRSynth annotation tag.

RP Report

This tool takes as input a TAR collection of rpSBML files, extracts all the information contained within the models, and outputs a CSV file that summarises the results.

RP Global Score

After performing all the different types of analysis as previously described, this tool normalises the different values from each heterologous pathway, and performs a weighted sum function using the following attributes:

  • Number of steps of the pathway
  • Best UniProt ID Selenzyme score
  • Target molecule flux
  • Gibbs free energy

Each of these characteristics for the score contains a weight associated to it, that must be defined by the user. If not specified, the tool considers the above attributes of a pathway equally to generate a global score. To normalise the number of steps, the user must specify the “Maximal number of steps”, as is defined by the RetroPath2.0 node. The advanced options include the name of the heterologous pathway, using the default as “rp_pathway”, the name of the FBA objective as defined by the FBC package. The IP address also needs to be defined when using the web service version of the tool. Lastly, the user must define the number of models to return based on the global score calculated.

rpOptBioDes

This tool takes as input a TAR collection of rpSBML files, that contain for each heterologous reaction a ranked list of UniProt identifiers for the sequences, and the list of links to the DNA registry SynBioHub for the desired vector backbones, resistance cassette, and promoters. An optimal design of experiments is performed by using OptBioDes based on logistic regression analysis with an assumed linear model for the response. The library size n can be selected and should be above a minimal threshold depending on the number of combinatorial complexity of the library.

The experimental design can be evaluated through the provided diagnostics for D-efficiency, power analysis and relative prediction variance, allowing the optimal selection of library size.

The resulting design is provided as an SBOL collection containing the definition of each DNA component and the combinatorial library of constructs.
